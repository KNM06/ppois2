# Лабораторная работа №4: Клиент-серверное веб-приложение модели пылесоса

**Документация системы имитации работы пылесоса с веб-интерфейсом**

## 1. Общее описание

Данная программная система представляет собой клиент-серверное веб-приложение для моделирования работы бытового пылесоса. Проект является развитием Лабораторной работы №1 и демонстрирует архитектурные приемы построения пользовательского интерфейса, принципы работы по протоколу HTTP и паттерн MVC (Model-View-Controller).

Главной особенностью архитектуры является **строгое разделение бизнес-логики и интерфейса**: ядро системы (папка `core`) полностью изолировано и используется совместно как классическим консольным интерфейсом (CLI), так и новым веб-сервером на базе фреймворка Flask.

## 3. Описание модулей

### 3.1. Пакет core (Общий код)

Содержит независимую бизнес-логику системы. Код внутри `core` не содержит функций ввода-вывода (print/input) и ничего не знает о том, в какой среде он запускается (консоль или веб).
- **VacuumCleaner**: Главный класс-агрегатор, управляющий компонентами (`Motor`, `DustContainer`, `Filter`).
- **Room**: Модель помещения с логикой накопления пыли.
- **exceptions**: Иерархия пользовательских ошибок для обработки нештатных ситуаций (переполнение бака, износ фильтра).

### 3.2. Веб-интерфейс (app.py и templates/index.html)

Клиент-серверная часть приложения, построенная на базе микрофреймворка **Flask**.

- **Сервер (`app.py`)**: 
  - Обрабатывает HTTP-запросы от браузера (маршрутизация).
  - При GET-запросах на корень (`/`) рендерит HTML-шаблон, передавая в него текущее состояние объектов из `core`.
  - При POST-запросах (`/power`, `/clean`, `/room/add` и т.д.) принимает данные из HTML-форм, вызывает соответствующие методы бизнес-логики и сохраняет изменения в JSON-файлы.
- **Клиент (`index.html`)**: 
  - Разметка страницы, стилизованная с помощью CSS.
  - Использует шаблонизатор **Jinja2** для динамического вывода переменных Python и логической отрисовки элементов (циклы `{% for %}`, условия `{% if %}`).

### 3.3. CLI интерфейс (cli.py)

Сохраненный консольный интерфейс из Лабораторной работы №1. Доказывает независимость ядра программы: может работать с теми же JSON-файлами состояния параллельно с веб-сервером.

## 4. Ключевые архитектурные решения

### Разделение ответственности (Separation of Concerns)
Проект разделен на три слоя:
1. **Данные**: JSON-файлы.
2. **Логика (Model)**: `core/`.
3. **Представление и управление (View/Controller)**: `app.py` + HTML (для веба) и `cli.py` (для консоли).

### Паттерн PRG (Post/Redirect/Get)
Для предотвращения проблемы двойной отправки форм (Double Submit Problem) реализован паттерн PRG. После каждого POST-запроса, изменяющего состояние системы (например, очистка контейнера), сервер отвечает HTTP редиректом (`302 Found`), заставляя браузер сделать новый GET-запрос для обновления страницы.

### Всплывающие уведомления (Flash Messages)
Обработка исключений из ядра (например, попытка уборки при заполненном контейнере `ContainerFullException`) перехватывается сервером и передается пользователю в браузер через механизм сессионных cookie в виде всплывающих сообщений (Flash).

## 5. Инструкция по запуску

### Предварительные требования
Для работы веб-интерфейса необходимо установить фреймворк Flask:

    pip install Flask

### Запуск веб-приложения (Л/Р №4)
Находясь в корневой директории проекта, выполните команду:

    python app.py

После запуска сервера откройте браузер и перейдите по адресу: `http://127.0.0.1:5000`

### Запуск консольного приложения (Л/Р №1)
Для проверки работы общей бизнес-логики через терминал выполните:

    python cli.py

### Запуск тестов
Для запуска всех unit-тестов выполните:

    python -m unittest discover tests
